# 新老架构深度对比

## 0. 先纠正一个偏见

之前的简报过于简化，把老架构描述成"过度设计的空中楼阁"，这不公平。老架构有很多值得保留的洞察，尤其是在 Core 层设计、二开分级、部署形态上。两个方案的核心分歧**不在于好坏，而在于节奏和起步策略**。

---

## 1. 两套方案到底在哪里相同

把两份文档摊开对比，会发现**底层高度一致**：

| 维度 | 老架构 | 新架构 | 结论 |
|------|--------|--------|------|
| Core 用 C++ | 是 | 是 | **完全相同** |
| Core 不依赖 GUI | 是（明确要求） | 是（明确要求） | **完全相同** |
| Core 做成动态库 | 是（C ABI） | 是（libvxl_core） | **完全相同** |
| 不暴露 OpenCV/Open3D 底层类型 | 是（自有数据对象） | 是（vxl:: 类型） | **完全相同** |
| Python 绑定（pybind11） | 是 | 是 | **完全相同** |
| 结构光重建自研 | 是（但没展开） | 是（详细展开） | 新架构在这里更深入 |
| OpenCV + Open3D + ONNX Runtime | 是 | 是 | **完全相同** |
| PCB 作为首要场景 | 是（第一优先级） | 是（第一场景） | **完全相同** |
| Recipe/检测模板 | 是 | 是 | **完全相同** |
| 二开靠 Python | 是（核心手段） | 是（核心手段） | **完全相同** |

**结论：两个方案共享同一个 Core 内核。** 如果老架构也优先只做一个场景（PCB），两者在 Phase 1 阶段的实际编码工作几乎一样——都是在写 `libvxl_core`。

---

## 2. 真正的分歧在哪里

分歧集中在 **Core 之上的部分**：

### 2.1 应用层形态

| 维度 | 老架构 | 新架构 |
|------|--------|--------|
| **上层形态** | Studio（设计器）+ Runtime（执行器）分离 | 单体 Qt 应用（工程模式 + 运行模式） |
| **Studio 技术** | Tauri 2 + WebView + TS/React | 不存在（Qt 承担设计功能） |
| **Runtime 技术** | C++ 宿主 + Python/Dear PyGui | Qt 桌面应用 |
| **部署产物** | .asapp 包（Studio 导出，Runtime 加载） | Recipe JSON + 应用本体 |

**老架构的优势：**
- Studio/Runtime 分离是成熟的工业软件模式（VisionPro Designer/Runtime、PLC 编程/运行就是这个模式）
- .asapp 包机制让现场部署变成"拷贝 + 导入"，非常干净
- Runtime 不暴露设计功能，操作员看到的界面更简洁、更安全
- Dear PyGui 让 Python 二开工程师可以直接操作 GUI，自由度高

**老架构的风险：**
- Tauri + Rust + TS + React + Dear PyGui，技术栈多，每个都要有人维护
- Dear PyGui 是小团队项目，长期维护不确定
- Studio/Runtime 分离在 Phase 1 就要设计两套程序的通信协议
- 工业领域用 WebView 做 Studio 没有先例可参考

**新架构的优势：**
- 技术栈少（C++ + Python + Qt），招人容易，维护简单
- Qt 在工业视觉领域验证充分（HALCON、VisionMaster 都用）
- 一个程序，两种模式切换，开发速度快
- Phase 1 可以专注 Core，不需要同时设计两个应用

**新架构的风险：**
- Qt 界面不如 Web 灵活，后期做复杂编排界面会吃力
- 单体应用到一定规模后可能需要拆分（但那时也积累了足够经验）
- Python 二开只有 API，没有 GUI 操作能力（除非后期加 PySide6）

### 2.2 二次开发体系

| 维度 | 老架构 | 新架构 |
|------|--------|--------|
| **分级** | 三级（配置 / Python 脚本 / C++ 插件） | 两级（Python API + JSON / C++ 插件预留） |
| **Python 的 GUI 能力** | Dear PyGui 直接操作界面 | 无（纯 API） |
| **UI 定制** | Python 可动态创建面板 | 只能改参数，不能改界面 |
| **流程编排** | Pipeline/Node 系统 + JSON 描述 | Python 脚本直接编排 |

**老架构在这里确实更完整。** 三级体系（配置→脚本→插件）的划分很清晰，而且 Python 能直接操作 GUI 对交付工程师来说价值很大。新架构说"Python 就是最好的编排语言"没错，但忽略了一个现实：**很多交付工程师能看懂 GUI 编排，写不了 Python 脚本。**

### 2.3 模型管理

**老架构有 models/ 目录和 .asapp 中的 models/ 结构，新架构确实遗漏了。**

工业检测软件中 AI 模型（异常检测模型、分类模型、分割模型）是重要资产：
- 需要版本管理
- 需要和 Recipe 绑定
- 需要支持热更新
- 部署时需要随项目打包

这一点老架构考虑得更周到。

### 2.4 扩展预留

| 维度 | 老架构 | 新架构 |
|------|--------|--------|
| 插件系统 | plugin_api（C ABI），有设计 | plugin_api.h 预留，无设计 |
| 机器人引导 | guidance 模块，有位置 | guidance.h 预留 |
| 远程通信 | gRPC/sidecar，有方案 | transport.h 预留 |
| Pipeline 引擎 | 有设计（Node/Port/Param） | 无（Python 替代） |
| 设备抽象 | ICamera2D/ICamera3D/ITrigger 完整 | camera.h 有，但没展开接口层次 |

老架构在扩展性上明显想得更远。

## 3. 重新评估：如果老架构也只做一个场景

假设老架构也采用"先做 PCB，只做 Core"的策略，两者的差异会缩小到什么程度？

### Phase 1 完全相同

两个方案的 Phase 1 都是写 `libvxl_core`：
- 相机采集
- 结构光重建
- 标定
- 高度图处理
- 3D 检测算子
- Python 绑定

**代码一行不差。**

### Phase 2 开始分叉

| | 老架构 Phase 2 | 新架构 Phase 2 |
|---|---|---|
| 做什么 | Qt 临时界面（先不做 Studio/Runtime 分离） | Qt 正式界面 |
| 实际工作量 | 类似 | 类似 |
| 区别 | 心理上知道这是"临时方案"，以后要拆 | 认为这就是最终方案 |

如果老架构在 Phase 2 也务实地先做一个 Qt 应用（而不是同时搞 Tauri + Dear PyGui），**那两者在 Phase 1-2 的实际执行完全一样**。

### Phase 3-4 才真正分叉

到了要做"多客户交付、二开体系、部署方案"的时候：
- 老架构走 Studio/Runtime + .asapp
- 新架构继续在 Qt 单体上加功能

**这才是真正需要做选择的时刻，而不是现在。**

---

## 4. 老架构真正好的部分

### 4.1 Studio/Runtime 分离的思路

这个想法本身是对的。工业软件确实需要区分"工程师调试环境"和"操作员运行环境"。新架构用"同一程序两个模式"替代，在早期够用，但到了大客户交付时可能不够：

- 客户可能不希望操作员看到工程模式的入口
- 运行环境可能是工控机，不需要装完整的开发环境
- 多工位部署时，每台机器装完整 Studio 是浪费

### 4.2 .asapp 包机制

"把项目打成一个包，Runtime 加载执行"这个设计非常优雅：
- 版本管理清晰
- 部署标准化
- 可以做签名/加密保护 IP
- 模型、脚本、参数、UI 定义全在一起

新架构的 Recipe JSON 只覆盖了参数部分，缺少模型、脚本、资源的打包。

### 4.3 统一数据对象

老架构定义了完整的数据对象体系：
```
ImageFrame / DepthMap / HeightMap / PointCloudData
MeshData / Pose6D / DefectRegion / InspectionResult
```

新架构也有类似定义（`vxl::` 类型），但没有老架构那么系统化。

### 4.4 设备抽象层

老架构的 `ICamera2D / ICamera3D / IProjector / ITrigger / IIO / IRobot / IPLC` 接口体系比新架构更完整。当你需要接多种相机或多种 PLC 时，这层抽象很关键。

### 4.5 VLM Copilot 定位

老架构对 VLM 的定位（辅助建站、辅助标注、辅助调参、辅助报告）很准确，不是让 VLM 做判定，而是做 Copilot。新架构直接说"推迟"，但其实可以在架构上预留接口而不实现。

---

## 5. 新架构真正好的部分

### 5.1 结构光重建的深度

新架构花了大量篇幅讲重建管线（相位计算、展开、材质适应性、GPU 加速），这些在老架构中一笔带过。**这是对的——重建管线才是技术核心，比 GUI 框架选型重要得多。**

### 5.2 务实的场景选择

新架构把 PCB/SMT 作为明确的第一场景，并解释了为什么（IPC 标准、市场大、复购强）。老架构把 PCB/塑料件/金属件/晶圆都列为第一优先级，边界太宽。

### 5.3 竞品分析视角

新架构指出"你的对手不是 HALCON，是卖相机+软件整体方案的厂商"。这个判断更准确，直接影响产品定位和定价策略。

### 5.4 执行节奏感

新架构给了清晰的 Phase 分期和具体的 checklist，可以直接开始执行。老架构更像"架构愿景"，缺少"明天开始写什么代码"的指引。

---

## 6. 综合建议：取两者之长

**执行策略用新架构（先做产品，聚焦 Core），架构视野用老架构（保留扩展设计）。**

具体来说：

### 从老架构吸收到新架构中的内容

| 要吸收的 | 怎么做 |
|---------|--------|
| **models/ 目录** | 新架构补上，在 recipes/ 同级或内部加 models/ |
| **设备抽象接口** | `core/include/vxl/` 中加 `device.h`，定义 ICamera/ITrigger/IIO 接口族 |
| **统一数据对象体系** | `types.h` 中系统化定义，参考老架构的命名 |
| **.asapp 包概念** | 当前先用 Recipe 目录结构，但按 .asapp 的思路组织（manifest + params + models + scripts） |
| **VLM 接口预留** | `core/include/vxl/` 中加 `vlm.h` 预留头文件 |
| **Studio/Runtime 分离意识** | Qt 应用中从一开始就把"工程逻辑"和"运行逻辑"分成独立模块，为将来拆分做准备 |

### 保持新架构的部分

| 保持的 | 理由 |
|--------|------|
| **技术栈精简**（C++ + Python + Qt） | 执行力 > 架构美感 |
| **Phase 1 聚焦 Core** | 重建管线才是核心 |
| **PCB 单场景切入** | 边界清晰才能做深 |
| **Python API 作为主要二开方式** | 最少基础设施，最大灵活性 |
| **不做 Pipeline/Node 引擎** | 等真正需要时再做 |

---

## 7. 一句话结论

> **两个方案共享同一个 Core，分歧在 Core 之上。Phase 1-2 执行用新架构（简单、快），Phase 3+ 的扩展视野参考老架构（完整、远）。老架构不是废案，是路线图。**
